## 一、 系统架构蓝图与部署链路

本项目背离了“每个项目弄个云主机”的传统打法，创新性地运用了基于 Vercel 上的 **Monobuild (主代码库统一单体构建) 体系**。以此解决独立代码互相拉扯导致的跨域、样式不统一或难以分享下载的痛点。

### 1.1 两个核心代码库的职能划分
1. **主枢纽仓库（私有库 - `my-website`）**
   - 它是承载 Vercel 自动化部署的唯一实体。
   - 根目录下存放着原生的核心外壳页面（如 `index.html`, `WORKING_MEMORY.md`, 后端 `/api` 文件夹）。
   - 同时将每一个要开发的工具（例如 `infiniteskill`）在本地作为它的一个纯粹的“子文件夹”挂载在内部一并开发。

2. **外部开源镜像库（公开库 - 如 `yuanbw2025/infiniteskill`）**
   - 它是主库里的子工具成熟后的影子切片。通过部分推进手段把 `infiniteskill/` 直接镜像推送到这，专供网站访客点击“下载源码”跳转过去白嫖或提 PR，完全**不参与**实际云端部署。

### 1.2 物理层级与构建流（生命周期）
当主代码被推送到 GitHub 的 main 分支后，触发的 Vercel Pipeline（构建管道）如下运作：

- **第一阶段（读取环境）**：识别读取 `vercel.json` 规则链和 `SECRET_GEMINI_KEY` 等关键参数。
- **第二阶段（拦截控制）**：跑自己手写的 Node 脚本 `build.mjs`。
  - 第一步它会进入特定的子目录层级（目前是 `infiniteskill/`）。
  - 对内部发起完整的 `npm run build` 工作流产生生产级原生 SPA `dist/` 数据包。
- **第三阶段（聚合归拢）**：
  - 销毁旧的根级 `public` 文件夹并新建。
  - 毫不留情地把原生首页 `index.html` 外壳丢入 `public`。
  - 再将所有工具打包好的 `dist/` 里的资源，整整齐齐地平移进 `public/infiniteskill/` 等对应预留路径槽位。
- **最终形态呈现**：部署器将这个拼合完整的 `public` 送上前台。当访客敲下 `https://yuanbw.vercel.app/infiniteskill/` 时，配合事先挂载在 `vercel.json` 里的无缝路由劫持重写机制，React SPA 顺利渲染。

## 二、 关键技术栈清单

| 生态模块 | 所用技术流派 | 技术选取目的及约束 |
|---------|------------|------------------|
| **主网站大门页** | 核心原生 `HTML5` + `Vanilla CSS` + 原生 JS | 图的是绝对的闪电加载跟极致的沉浸动态极客美学，无需任何框架开销。|
| **后端防盗刷代理** | `Vercel Serverless Function` (`Node.js`) | 直接屏蔽了 Web 端暴漏主密匙的安全事故。目前代码结构极为严谨，强制 `process.env` 获取令牌并主动阻截一切空环境。|
| **智能编译器子项** | `Vite 6` + `React 19` + `TailwindCSS v4` | 主力功能工具，利用组件化高速渲染业务组件。|
| **AI 模型驱动层** | `Google Gemini API v1beta (Native)` | 为了避免兼容层不稳定，所有后台请求直通 Google 底部原生协议。并手动将响应降维打击回前端熟悉的 OpenAI 结构。|
